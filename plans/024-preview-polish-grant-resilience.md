## Slices

### G1 — Backend-lost toasts on permission grant (P0)

The renderer must treat a dropped backend WS as the transient, self-healing
state it is (the offline badge already does — this is the toast path missing
the same discipline):

1. In `reportError` (use-studio.tsx), classify "Backend WebSocket is not
   connected." / "Backend socket is not connected." as **transient**: keep
   one keyed toast (`id: 'backend-disconnected'`) that a reconnect dismisses,
   never a stack; better, suppress it entirely while `wsStatus !==
   'connected'` since the Session badge already shows "Backend offline"
   (backend-resilience smoke).
2. The grant flow (`media-access.ts` → renderer) should quiesce the meter/
   preview pollers for the known restart window instead of letting them fire
   into a closing socket — set a "reconnecting" gate the moment a grant
   reports `restarted: true`, cleared on `backend.ready`.
3. Pure helper `isTransientBackendError(message)` + unit test; the audio-meter
   and device-refresh catch paths route transient errors to the keyed toast.

**Done when**: a media-access grant that restarts the backend shows at most
one calm "Reconnecting…" state (or nothing — the badge covers it), never a
stack of red; a REAL backend failure still surfaces. Unit test on the
classifier; `pnpm --filter @videorc/desktop test`.

### G2 — Support bundle reports the real app version + commit (P1)

The bundle is generated by the backend, which only knows its own crate
version. The DESKTOP app version + git commit must reach it:

1. Main passes `app.getVersion()` (Electron `apps/desktop/package.json`
   version) and the build commit to the backend at bundle-export time (the
   IPC already carries a request; add `appVersion` + `commit`), OR the export
   command stamps them main-side after the backend returns its JSON.
2. `SupportBundleApp` gains `appVersion` (the 0.9.16 value) and `commit`
   alongside the existing backend `version`; the redactor leaves both.
3. Bump `schemaVersion` to 2; note the field split.

**Done when**: a fresh bundle shows the packaged app version (not 0.9.0) and
a commit; `cargo test -p videorc-backend support_bundle`, verifier +
`pnpm test:scripts` if the verifier asserts shape.

### G3 — Preview start-stretch (P2, native preview)

The CSS box is already output-aspect-locked; the stretch is the native
surface presenting at drawable bounds that differ for the first frames.
Per the sizing oracle:

1. Hold the native surface HIDDEN (fallback slot showing, output-aspect) until
   the first real frame's drawable matches the committed output aspect within
   tolerance — never present a mid-conform frame into the slot.
2. Verify `surfaceWindowPlacement` derives the surface rect from the OUTPUT
   aspect, not the source aspect (Cam Link's 3840x2160 vs a non-16:9 output
   is the stretch vector); the drawable must letterbox the source into the
   output rect, not fill-stretch it.
3. Extend `probe:preview-lifecycle` (or the sizing probe) to assert the
   presented drawable aspect == output aspect from the first presented frame
   across a source/output aspect MISMATCH (e.g. 15.5:9 screen → 16:9 output).

**Done when**: opening the preview with a source whose aspect differs from the
output shows no stretch at any point (fallback → native hand-off is aspect-
stable); the probe asserts it; owner by-eye on the Cam Link.

### G4 — Click-flash wait hint (P2)

On a HEALTHY, presenting preview, a click must not surface the wait-detail
hint:

1. The presenting watch (plan 021 F1) should only push a wait-detail string
   when `kind` is `heal`/`stalled` — a `presenting`/`observing` tick on a
   healthy surface must leave the hint CLEARED, and a focus/click re-kick must
   not reset it to the default "Waiting for preview" copy.
2. Trace the click path: `showInactive`/placement on focus re-runs the status
   build (index.ts ~3484) — ensure it doesn't momentarily flip
   `firstFrameContract` back to `pending` and repaint the hint.
3. Regression: a presenting-watch tick sequence `presenting → (click re-kick)
   → presenting` never emits a non-empty wait-detail (decision-core unit
   test).

**Done when**: clicking a live preview shows no flashing text; unit test on
the decision core; owner by-eye.

## Non-negotiables

- An EXPECTED backend restart never renders as stacked error toasts — the
  offline badge is the single source of truth for connection state.
- The support bundle must always identify the shipped app version — remote
  triage depends on it (this whole plan nearly mis-triaged on "0.9.0").
- No preview frame is ever presented mid-conform (the stretch is a hand-off
  discipline bug, not a resize bug).
